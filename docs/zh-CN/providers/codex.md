# Codex

本指南面向希望在 T3 Code 中使用多个 Codex 账户的用户。

常见原因：

- 工作项目使用工作账户
- 个人项目使用个人账户
- 一个账户达到限额时切换到另一个账户
- 共享一份 Codex 历史记录，而不是维护两套完全独立的 Codex 配置

## 我只使用一个 Codex 账户

使用默认供应商。

在设置中，Codex 供应商可以保持如下配置：

```text
显示名称：Codex
CODEX_HOME 路径：~/.codex
影子主目录路径：留空
```

按常规方式登录 Codex：

```bash
codex login
```

## 我想使用工作和个人 Codex 账户

使用一个真实 Codex 主目录和一个影子主目录。

推荐配置：

```text
~/.codex      共享 Codex 主目录
~/.codex_p    第二个账户的认证信息
```

思路如下：

- 两个账户都能看到相同的 T3/Codex 会话
- 每个账户保留自己的登录信息
- 现有会话可以使用任一账户继续

### 配置第一个账户

按常规方式登录：

```bash
codex login
```

这是 `~/.codex` 使用的账户。

在 T3 Code 设置中为它使用明确的名称：

```text
显示名称：Codex Work
CODEX_HOME 路径：~/.codex
影子主目录路径：留空
```

### 配置第二个账户

使用独立的 Codex 主目录登录：

```bash
mkdir -p ~/.codex_p
CODEX_HOME=~/.codex_p codex login
```

在 T3 Code 设置中添加另一个 Codex 供应商：

```text
显示名称：Codex Personal
CODEX_HOME 路径：~/.codex
影子主目录路径：~/.codex_p
```

关键在于两个供应商使用同一个 `CODEX_HOME` 路径，只有第二个供应商设置了`影子主目录路径`。

## 我正在使用哪个账户？

打开设置并查看供应商行。

对于能够报告账户电子邮件的供应商，T3 Code 会显示已认证的电子邮件。电子邮件默认模糊显示；点击模糊的电子邮件即可查看。

使用显示名称和强调色，可以在模型选择器中轻松区分账户。

## 我需要不同的 API 密钥或端点

使用设置中该供应商的`环境变量`区域。

当兼容 Codex 的配置需要账户专用变量时，这很有用。将变量添加到应接收它们的供应商实例，并将 API 密钥或令牌标记为敏感。敏感值会作为服务器机密存储，保存后不会再发送回应用。

## 可以在现有会话中切换账户吗？

可以，前提是两个 Codex 供应商共享相同的 `CODEX_HOME` 路径。

例如：

```text
Codex Work      CODEX_HOME 路径：~/.codex
Codex Personal  CODEX_HOME 路径：~/.codex，影子主目录路径：~/.codex_p
```

这两个供应商被视为可兼容继续，因此锁定的模型选择器可以同时显示它们。

如果添加第三个使用完全不同 `CODEX_HOME` 路径的 Codex 供应商，T3 Code 会将其视为不同的工作区。它不会出现在 `~/.codex` 下创建的现有会话中。

## 如果两个账户看起来相同

如果两个 Codex 供应商显示同一账户或相同的异常模型列表：

1. 检查设置中的电子邮件。
2. 刷新供应商状态。
3. 确认第二个供应商已设置`影子主目录路径`。
4. 确认影子目录拥有自己的 `auth.json`。
5. 如果曾将 `~/.codex` 复制到影子目录，请删除除 `auth.json` 外的所有内容。

清理示例：

```bash
find ~/.codex_p -mindepth 1 ! -name auth.json -exec rm -rf {} +
```

## 何时使用独立的 CODEX_HOME

仅在希望使用独立 Codex 工作区时，才使用完全独立的 `CODEX_HOME` 路径。

这意味着会话彼此独立，并且在旧会话中切换账户的能力较弱。大多数双账户用户应使用共享主目录加影子主目录的配置。
