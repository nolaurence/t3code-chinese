# 远程架构

本文档描述 T3 Code 中一等远程环境的目标架构。

本文有意以架构为先，不定义完整实施计划或面向用户的发布清单。目标是建立核心模型，使远程支持能够在不进行又一次大范围重写的情况下加入。

## 目标

- 将远程环境作为一等产品原语，而不是特殊情况。
- 支持通过多种方式访问同一环境。
- 保持 T3 服务器作为执行边界。
- 让桌面端、移动端和 Web 端共享同一概念模型。
- 除非产品压力证明必要，否则不引入本地控制平面。

## 非目标

- 使用自定义传输协议替换现有 WebSocket 服务器边界。
- 将 SSH 作为唯一的远程方案。
- 在计算机之间同步供应商认证。
- 在首次迭代中提供所有访问方式。

## 高层架构

T3 已经拥有清晰的运行时边界：客户端通过 HTTP/WebSocket 与 T3 服务器通信，服务器负责编排、供应商、终端、Git 和文件系统操作。

远程支持应保留该边界。

```text
┌──────────────────────────────────────────────┐
│ 客户端（桌面端/移动端/Web）                 │
│                                              │
│ - 已知环境                                   │
│ - 连接管理器                                 │
│ - 感知环境的路由                             │
└───────────────┬──────────────────────────────┘
                │
                │ 解析一个访问端点
                │
┌───────────────▼──────────────────────────────┐
│ 访问方式                                     │
│                                              │
│ - 直接 ws / wss                              │
│ - 隧道 ws / wss                              │
│ - 桌面管理的 SSH 引导和转发                  │
└───────────────┬──────────────────────────────┘
                │
                │ 连接到一台 T3 服务器
                │
┌───────────────▼──────────────────────────────┐
│ 执行环境 = 一台 T3 服务器                   │
│                                              │
│ - 环境身份                                   │
│ - 供应商状态                                 │
│ - 项目/会话/终端                             │
│ - Git/文件系统/进程运行时                    │
└──────────────────────────────────────────────┘
```

关键决策是：远程性在环境连接层表达，而不是通过拆分 T3 运行时表达。

## 领域模型

### ExecutionEnvironment

`ExecutionEnvironment` 是一个正在运行的 T3 服务器实例。

它负责：

- 供应商可用性和认证状态
- 模型可用性
- 项目和会话
- 终端进程
- 文件系统访问
- Git 操作
- 服务器设置

它由稳定的 `environmentId` 标识。

这是跨客户端共享的原语。桌面端、移动端和 Web 端都应以相同概念理解它。

### KnownEnvironment

`KnownEnvironment` 是客户端侧保存的条目，表示客户端知道如何访问的环境。

它不是由服务器创建的，而是本地存储在设备或客户端配置中。

例如：

- 已保存的局域网 URL
- 已保存的公开 `wss://` 端点
- 由桌面管理的 SSH 主机条目
- 已保存的隧道环境

已知环境在首次成功连接前可能知道目标 `environmentId`，也可能不知道。

在托管 Web 应用中，已知环境存储在浏览器本地。托管配对 URL 可以创建保存条目，但不会为托管应用提供服务器端控制平面或会话状态副本。

### AccessEndpoint

`AccessEndpoint` 是访问一个已知环境的具体方式。

这是避免 SSH 主导整个模型的关键抽象。

一个环境可以拥有多个端点：

- `wss://t3.example.com`
- `ws://10.0.0.25:3773`
- 隧道 Relay URL
- 由桌面管理的 SSH 隧道，它会解析到本地转发的 WebSocket URL

环境保持不变，变化的只有访问路径。

### AdvertisedEndpoint

`AdvertisedEndpoint` 是服务器或桌面端提供的环境候选端点。后端通过它告知客户端哪些 URL 可能适合配对和重新连接。

`AdvertisedEndpoint` 有意比完整访问模型更窄：

- 描述一组具体的 HTTP 和 WebSocket 基础 URL
- 可以将端点标记为默认、可用或不可用
- 包含回环、局域网、私有、公开或隧道等可达性提示
- 包含端点能否从托管 HTTPS 应用使用等兼容性提示

客户端应将发布的端点视为提示，而不是从当前设备可用的证明。最终连接尝试仍决定端点是否可达。

UI 在网络访问摘要中显示一个默认发布端点，其余端点收纳在可展开的高级列表中。默认端点控制配对二维码和主要复制操作。用户可以覆盖默认值，但该覆盖值是 UI 偏好，而不是后端配置。

只要可能，都应按稳定的端点类型而不是原始 URL 持久化覆盖值。例如，局域网端点应存储为桌面局域网端点偏好，而不是 `192.168.x.y`，因为用户切换网络时地址可能变化。供应商端点应使用供应商专用的稳定键，例如 Tailscale IP 或 Tailscale MagicDNS HTTPS。自定义端点可以回退到具体身份。

未保存用户默认值时，端点选择应优先：

1. 与托管 HTTPS 应用兼容的端点
2. 明确标记为默认的端点
3. 非回环端点
4. 仅对同机客户端使用回环端点

这样可以集中端点发现，同时避免让 Tailscale 或未来隧道服务等某个供应商成为核心环境模型的一部分。

### 端点供应商

端点供应商是为当前环境提供发布端点的附加组件。

供应商边界有意放在核心环境模型之外：

- 核心负责 `ExecutionEnvironment`、已保存环境、配对和连接生命周期
- 供应商发现或生成端点
- 供应商返回规范化的 `AdvertisedEndpoint` 记录
- UI 和配对逻辑从这些记录中选择，无需了解供应商专用命令

第一个供应商是 Tailscale。它可以从本机发现 Tailnet IP 和 MagicDNS 地址，并将其发布为额外的端点候选。未来的供应商（例如托管隧道服务）应接入相同结构，而不是添加独立的远程环境路径。

供应商专用的可信度仍应只是提示。客户端只有在 Tailscale 端点成功建立浏览器或桌面连接后，才能将其视为已连接。

### 托管配对请求

托管配对请求是静态 Web 应用的引导 URL，而不是传输方式。

示例：

```text
https://app.t3.codes/pair?host=https://backend.example.com:3773#token=PAIRCODE
```

托管应用读取 `host` 参数和配对令牌，直接与该后端交换令牌，再将最终环境记录保存到浏览器本地存储。

重要约束：

- 托管应用不代理 HTTP 或 WebSocket 流量
- 浏览器仍必须能够直接访问后端
- HTTPS 页面只能连接 HTTPS/WSS 后端
- HTTP 局域网端点应继续使用直接桌面或 CLI 配对 URL
- 令牌必须放在 URL 哈希中，避免发送到托管应用源

### RepositoryIdentity

`RepositoryIdentity` 仍是跨环境尽力而为的逻辑仓库分组机制。

它不用于路由，只用于 UI 分组，以及关联同一仓库的本地和远程克隆。

### 工作区/项目

当前 `Project` 模型仍为环境本地模型。

这意味着：

- 本地克隆和远程克隆是不同项目
- 它们可以共享一个 `RepositoryIdentity`
- 会话仍绑定到一个环境中的一个项目

## 访问方式

访问方式只回答一个问题：

客户端如何通过 WebSocket 与 T3 服务器通信？

它们不回答：

- 服务器如何启动
- 谁管理服务器进程
- 环境是本地还是远程

### 1. 直接 WebSocket 访问

示例：

- `ws://10.0.0.15:3773`
- `wss://t3.example.com`

这是基础模型，应作为一等默认值。

优点：

- 适用于桌面端、移动端和 Web 端
- 不需要客户端专用的进程管理
- 最适合托管或自行管理的远程 T3 部署

浏览器安全规则是该访问方式的一部分。托管 HTTPS Web 客户端可以连接 `wss://` 后端，但不能连接普通 `ws://` 或 `http://` 局域网后端，因为这会产生混合内容。

### 2. 隧道 WebSocket 访问

示例：

- 公开 Relay URL
- 私有网络 Relay URL
- pipenet 等本地隧道产品

从客户端角度看，这仍是直接 WebSocket 访问，区别在于路由由隧道或 Relay 中介。

对于 T3，最好将隧道建模为另一个 `AccessEndpoint`，而不是不同种类的环境。

这尤其适用于：

- 主机位于 NAT 后方
- 入站端口不可用
- 移动端需要访问桌面托管的环境
- 需要访问计算机但不希望暴露原始局域网或公开端口

从架构上看，Tailscale 支持的访问位于此处，尽管当前实现是端点发现而不是 T3 管理的隧道。它提供私有网络端点，并让现有 HTTP/WebSocket 客户端路径执行实际连接。

### 3. 桌面管理的 SSH 访问

SSH 是访问和启动辅助工具，不是独立的环境类型。

桌面主进程可以使用 SSH：

- 访问计算机
- 探测计算机
- 启动或复用远程 T3 服务器
- 建立本地端口转发

之后，渲染器仍应使用普通 WebSocket URL 连接转发后的本地端口。

这样可使渲染器传输模型与其他所有访问方式保持一致。

桌面主进程负责 SSH 桥接，因为它能生成本地 SSH 进程、管理 askpass 提示、写入临时启动脚本并清理转发。渲染器收到已保存的环境记录并通过转发 URL 连接；普通环境流量不应需要 SSH 专用 RPC 路径。

## 启动方式

启动方式回答另一个问题：

T3 服务器如何在目标计算机上出现？

设计中应保持启动与访问相互独立。

### 1. 已存在的服务器

最简单的启动方式是不启动。

用户或运维人员已经在目标计算机上运行 T3，客户端通过直接或隧道 WebSocket 端点连接。

这应是首个发布的远程模式，因为它能以最少的额外机制验证环境模型。

### 2. 桌面通过 SSH 管理远程启动

这是 Zed 最适合作为参考的位置。

可借鉴 Zed 的思路：

- 远程探测
- 平台检测
- 带有 PID/日志元数据的会话目录
- 便于重新连接的启动器行为
- 桌面端负责连接 UX

T3 应有所不同的地方：

- 渲染器和远程运行时之间不使用自定义标准输入输出/套接字代理协议
- 不尝试让远程运行时表现为编辑器传输
- 保持最终客户端到服务器的连接为 WebSocket

推荐的 T3 流程：

1. 桌面端通过 SSH 连接。
2. 桌面端探测远程计算机并验证 T3 可用性。
3. 桌面端启动或复用远程 T3 服务器。
4. 桌面端建立本地端口转发。
5. 渲染器将转发的 WebSocket 端点作为普通环境连接。

已保存环境应记录它由桌面 SSH 启动创建，但该信息只用于重新连接和生命周期 UX。此元数据不应改变服务器协议或环境身份模型。

失败处理应明确：

- SSH 认证失败应在保存任何环境之前显示
- 远程启动失败应尽可能包含远程日志或启动器命令输出
- 转发端口失败应使已保存环境保持断开，而不是回退到无关端点
- 重新连接时，应先尝试恢复 SSH 桥接，再重新连接普通 WebSocket 客户端

### 3. 客户端管理的本地发布

这是远程启动的反向操作：本地 T3 服务器已经运行，客户端通过隧道发布它。

适用于：

- 将桌面托管的环境暴露给移动端
- 无需更改路由器或防火墙设置即可提供临时远程访问

这仍是启动问题，而不是新的环境种类。

## 为何访问和启动必须分离

这两个问题很容易混淆，但分离它们可以防止架构漂移。

例如：

- 手动托管的 T3 服务器可以通过直接 `wss` 访问。
- 同一服务器也可以通过隧道访问。
- SSH 管理的服务器可以通过 SSH 启动，再通过转发的 WebSocket 访问。
- 本地桌面服务器可以通过隧道发布给移动端。

在所有这些情况下，`ExecutionEnvironment` 都是同一种对象。

只有启动和访问路径不同。

## 安全模型

远程支持必须假设某些环境可通过不可信网络访问。

这意味着：

- 支持远程的环境应要求显式认证
- 隧道暴露不应依赖隐蔽性
- 客户端保存的端点应携带足够的认证元数据以安全地重新连接

T3 服务器已经支持 WebSocket 认证令牌。它应成为环境访问的一等部分，而不是继续作为偶然的查询参数约定。

对于公开可访问的环境，应将认证访问视为必需。

托管配对应仅被视为客户端便利功能。托管应用不得通过查询参数接收配对令牌、不得在服务器端存储配对状态，也不得暗示 HTTP 后端在 HTTPS 浏览器上下文中安全或可达。

## 与 Zed 的关系

Zed 是管理式远程启动和重新连接行为的有用参考实现。

相关经验包括：

- 远程引导应明确
- 重新连接应是一等能力
- 连接 UX 属于客户端外壳
- 运行时所有权应明确保留在远程主机上

关键差异在于传输形态。

Zed 需要自定义代理/服务器协议，因为它的远程边界位于编辑器和项目运行时之下。

T3 不应复制这一部分。

T3 已经拥有正确的运行时边界：

- 每个环境一台 T3 服务器
- 客户端和环境之间使用普通 HTTP/WebSocket

因此 T3 应借鉴 Zed 的启动纪律，而不是其传输协议。

## 推荐发布顺序

1. 一等的已知环境和访问端点。
2. 直接 `ws` / `wss` 远程环境。
3. 已认证的隧道环境。
4. 桌面管理的 SSH 启动和转发。
5. 在基础运行时路径得到验证后改进多环境 UI。

该顺序保持架构网络优先且与传输无关，同时仍为更丰富的管理式远程流程留出空间。
