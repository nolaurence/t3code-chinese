T3 Connect 控制平面与托管端点流程

T3 Connect 将本地已授权的 T3 环境与已登录的云端用户关联，为该环境预置托管 HTTPS/WSS 端点，并代理受证明约束的远程连接。本地环境仍是签发环境访问凭据的权威。 托管 Relay 存储关联、协调托管端点资源、验证签名证明并发送智能体活动通知。

客户端 -> Relay：Clerk Bearer 或 Relay DPoP Relay -> 环境：Relay 签名的健康和签发证明 环境 -> Relay：环境 Bearer 和已签名活动证明 客户端 -> 环境：环境访问令牌和 DPoP

安全不变量

云端身份不是环境登录。远程访问需要同时满足以下所有条件：

  • 通过 Clerk 认证的 T3 Connect 用户；
  • 该用户与环境之间存在活动 Relay 关联；
  • 远程客户端持有 DPoP 证明密钥；
  • 已关联本地环境接受 Relay 签名的签发请求；以及
  • 环境签发的凭据绑定到远程客户端的 DPoP 指纹。
Relay 无法自行签发环境访问令牌。它只能请求本地环境为已授权云端用户签发短期引导凭据。

在正常连接流程中，Relay 无法冒充远程客户端：

  • 普通环境 API 流量从远程客户端直接经过托管端点。Relay 不会收到最终环境访问令牌。
  • 一次性引导凭据和最终环境访问令牌都绑定到远程客户端的 DPoP 公钥指纹。兑换或使用它们 需要由对应私钥签名的请求证明，而私钥始终保留在客户端。
  • Relay 签名的健康和签发证明具有短期、受众绑定、作用域限制和重放保护。只有专用云端点 接受这些证明，普通环境 API 不会将其视为凭据。
  • 环境健康和签发响应由已关联环境签名并绑定请求 nonce，因此隧道后的其他进程无法冒充 已关联环境。
托管 Relay 仍是受信任的签发代理：它持有云端签发私钥。DPoP 会阻止 Relay 或隧道传输 在正常流程中复用凭据，但 Relay 签名权威一旦失陷仍会造成危害。

产品边界

通用本地连接器称为 Relay 客户端。当前实现层使用 cloudflared，但它是内部传输细节，而不是产品合约。

  • Relay Worker 负责云端关联、托管端点分配和通知。
  • 环境服务器负责本地会话、环境密钥和访问令牌。
  • Relay 客户端只暴露环境的回环 HTTP 服务器。
  • 引导完成后，Web 和移动客户端直接连接托管环境端点。

托管端点协调

托管端点资源以 (userId, environmentId) 为键。Relay 在预置前保留稳定的 主机名和隧道名，然后在 Postgres 中记录隧道 ID、DNS 记录 ID 和就绪状态检查点。

1. 保留确定性分配 主机名和隧道名由 Relay 阶段、云端用户和环境 ID 派生。
2. 复用或创建隧道 复用已有命名隧道。隧道入口被重写到已验证的回环源。
3. 协调 DNS 更新已有 CNAME 并移除重复记录。创建冲突通过列出并协调最终生效记录来恢复。
4. 标记为就绪 连接和状态请求只使用主机名仍与 Relay 管理命名空间匹配的就绪分配。

取消关联清理

Relay 取消关联会先移除用户的托管端点分配，再撤销用户与环境的关联：

  • 删除已有的托管 DNS 记录；
  • 删除已有的 Cloudflare 隧道；
  • 移除分配行；
  • 撤销用户的环境关联；以及
  • 仅在该环境密钥的最后一个活动关联消失后，撤销环境发布凭据。

本地取消关联会停止 Relay 客户端，并移除本地持久化的 Relay URL、签发者、已关联用户、 环境发布凭据、云端签发公钥、连接器配置和智能体活动偏好。

Cloudflare 隧道运行配置

托管隧道是私有应用后端，而不是通用公开托管。Relay 为每个活动 (userId, environmentId) 分配预置一个命名 Cloudflare 隧道和一条 DNS 记录。

  • 生命周期：环境关联时创建隧道，在可安全重试的协调中复用，在关联存续期间 保留，并在取消关联时删除。如果拆除失败，会保留分配检查点以便重试清理。
  • 源：隧道入口限制为已关联环境服务器经过验证的回环 HTTP 源。拒绝任意上游 主机和原始 TCP 源。
  • 协议：托管端点只承载 HTTPS 请求/响应流量和 WSS 连接。远程客户端通过隧道 使用普通环境 HTTP API 和已认证 WebSocket RPC 传输。
  • Relay 生成的流量：托管 Relay 在状态检查和远程连接引导期间发送少量 HTTPS 健康检查和凭据签发请求，不代理稳定状态下的远程会话。
  • 流量形态：预期负载是交互式且由用户驱动。活动远程会话可能保持 WebSocket 打开并交换终端、智能体和 UI 更新；不活动的已关联环境应只有很少或没有隧道流量。
  • 容量规划：项目尚未发布，因此还没有生产带宽基线。隧道数量随活动的用户/环境 关联分配增长，带宽和并发 WebSocket 则随活动远程使用增长。

Relay 客户端安装

桌面 Web UI 可用性检查和安装使用本地 WebSocket RPC 方法： cloud.getRelayClientStatus 和流式 cloud.installRelayClient。安装流会发出当前阶段：检查、等待锁、下载、验证、 安装、校验和激活。

Web UI 在安装前请求确认，并显示自定义进度对话框。CLI 直接调用同一本地 Relay 客户端 服务，并在终端中报告相同的进度阶段。

Relay 基础设施

  • 用于托管 Relay HTTP API 的 Cloudflare Worker。
  • 名为 t3coderelay 的 PlanetScale Postgres 数据库。
  • 生产环境使用共享数据库，非生产阶段使用数据库分支。
  • 禁用缓存且限制源连接数的 Cloudflare Hyperdrive。
  • 用于托管端点协调的 Cloudflare 隧道和 DNS 绑定。
  • Cloudflare APNs 发送队列和死信队列。
  • 每五分钟清理过期 DPoP 重放行的计划任务。
  • 每个阶段的 Axiom OTLP 链路数据集、限定范围的摄取令牌和最近 span 视图。

部署配置

边界 配置
Relay 基础设施 RELAY_API_ZONE_NAME、RELAY_TUNNEL_ZONE_NAME、可选的 RELAY_DOMAIN, CLERK_PUBLISHABLE_KEY, CLERK_SECRET_KEY、CLERK_JWT_AUDIENCE 和 APNs 凭据。
源码构建的桌面/服务器云端功能 可选的 T3CODE_RELAY_URL、T3CODE_CLERK_PUBLISHABLE_KEY 和 T3CODE_CLERK_CLI_OAUTH_CLIENT_ID。发布构建在构建时注入公开值。
Web 和移动云端客户端 公开 Clerk 可发布密钥、Clerk JWT 模板名称和 Relay URL。Relay URL 必须规范化为绝对 HTTPS 源，并且不能包含凭据、查询、片段或非根路径。

实施清单

  • Relay 云端签发私钥只保留在托管端，只将公钥安装到环境中。
  • 将环境签名私钥保留在本地，并以原子方式持久化密钥对创建。
  • 关联或更改 Relay 配置前，要求本地 relay:write 会话。
  • 发送凭据前，将 Relay URL 验证为安全的绝对 HTTPS 源。
  • 关联证明边界两侧都只允许回环隧道源。
  • 从就绪分配行解析托管端点出站地址，而不是使用客户端提供的 URL。
  • 在 Relay 到环境的请求上禁用重定向。
  • 使用隧道响应前验证环境签名的响应证明。
  • 保持隧道和 DNS 预置可安全重试，并在取消关联时解除预置。
  • 仅在最后一个活动关联消失后撤销共享环境发布凭据。
  • 对外发布前，脱敏并限制本地智能体失败详情长度。

标准参考

  • RFC 8252：原生应用基于浏览器的 OAuth，由 CLI PKCE 授权流程使用。
  • RFC 8693：OAuth 令牌交换，用于从 Clerk 令牌交换到 Relay DPoP 令牌。
  • RFC 9449：DPoP 持有证明，用于 Relay 和环境访问令牌。
